# v7.1 — Pipeline of Sieves

> Each stage is a sieve. Each sieve is multiple parallel pathways.
> Each pathway is molded so only the correct function can emerge.
> All weights are ternary. Topology IS the type system. Wrong
> computations are unreachable, not forbidden.
>
> The mold carves itself during training.

## Architecture: from v7 to v7.1

**v7 (validated):** Single pipeline. One transformer per stage.
Superposition overloads at depth — interference kills arithmetic
accuracy and semantic generalization.

**v7.1 (proposed):** Pipeline of sieves. Multiple parallel pathways
per stage. Sieve pre-separates the superposition — each pathway
carries one type of computation. Interference eliminated by topology.

```
v7:   [S1] → [S2] → [S3] → [S4]     (single pathway per stage)

v7.1: [S1 sieve] → [S2 sieve] → [S3 sieve] → [S4 sieve]
       ├─ A          ├─ A          ├─ A          ├─ A
       ├─ B          ├─ B          ├─ B          ├─ B
       ├─ C          ├─ C          ├─ C          ├─ C
       └─ D          └─ D          └─ D          └─ D
```

## All-Ternary, 250M Parameters

Every weight in the model is ternary: {-1, 0, +1}.

```
+1: pass activation through (add)
-1: negate activation (subtract)
 0: disconnect (this connection doesn't exist)
```

**Ternary IS the mold material.** The 0 weights define the sieve
shape — which connections exist. The ±1 pattern defines the function —
what the pathway computes. The sparsity pattern IS the topology.
Different pathways crystallize different topologies during training.

**Activations stay float.** Attention scores (Q·Kᵀ), softmax, and
intermediate computations are float32. The Church encoding
superposition has full precision. Ternary weights route and select;
float activations carry the computation.

**Proven by BitNet b1.58:** ternary weights match float precision
at scale. 3B ternary ≈ 3B float16 on benchmarks.

### Parameter budget

```
250M ternary weights:
  Storage:    250M × 2 bits = 62.5 MB (fits in M3 Ultra L2 cache)
  Compute:    add/subtract only — no floating-point multiply in matmul
  Throughput: estimated 100K-200K tok/s on M3 Ultra
  Compare:    v7 mixed 27M = ~55K tok/s, ~40 MB
```

### Proposed dimensions

```
d_model = 1024 (Church encoding superposition headroom)
Vocab = 50277 (GPT-NeoX, keep for forward compatibility with Dolma)

Stage 1 SIEVE (Surface):      512 pos, 2L per pathway, 4 pathways
Stage 2 SIEVE (Structural):    64 pos, 3L per pathway, 4 pathways
Stage 3 SIEVE (Semantic):       8 pos, 4L per pathway, 4 pathways
Stage 4 SIEVE (Reasoning):      1 pos, 6L per pathway, 4 pathways

Embedding: 50277 × 1024 = 51.5M (ternary)
Per-stage: depends on pathway d_model, d_ff, interaction design
Reducers: between stages (cross-attention pooling)
Feedbacks: backward cascade through sieves
Total target: ~250M ternary
```

**Open:** d_model per pathway. Options:
- Each pathway gets full d_model=1024 (wider but fewer pathways)
- Each pathway gets d_model=256, 4 pathways = 1024 total (narrower but more pathways)
- Mixed: surface pathways narrow, deep pathways wide

## How Attention Works in the Sieve

### The fundamental cycle (per layer)

```
Attention:  β-reduce in superposition
            (smear functions into weighted blend of all possible applications)
FFN gate:   index into superposition
            (SwiGLU selects which computation is active)
FFN up/down: compute the indexed reduction
            (expand to d_ff workspace, apply, compress back)
→ new superposition for next layer
```

### Why sieve helps

Single pathway: ALL pending reductions smeared into ONE superposition.
FFN must index through crowded, interfering patterns. Accuracy limited
by interference floor.

Sieve: different reductions flow through DIFFERENT pathways. Each
pathway's superposition is sparse and clean. FFN index is trivial
because each pathway only carries its specialty.

```
Single pathway:  superposition = {add + multiply + compare + compose}  → interference
Sieve pathway A: superposition = {add}                                 → clean
Sieve pathway B: superposition = {multiply}                            → clean
Sieve pathway C: superposition = {compare}                             → clean
Sieve pathway D: superposition = {compose}                             → clean
```

### Church encoding capacity

Qwen3.5-35B-A3B proves 17-digit Church encoding arithmetic via
attention. Church encoding turns numbers into nested lambda functions.
Arithmetic becomes function composition = beta reduction = what
attention does natively.

At d_model=1024, the superposition workspace (d_ff=4096) is half
Qwen's width. With sieve separation reducing interference, v7.1
might approach Qwen's digit ceiling despite much smaller size.

## Training: Cone-Shaped Northstar

### The search space problem

250M ternary weights → 3^250,000,000 possible topologies. Intractable
without structure. The cone provides structure.

### Relational loss (pathway differentiation)

Each pair of pathways within a stage is penalized for redundancy.
If pathway A and pathway B learn the same function, loss increases.
Forces specialization — each pathway MUST find a different function.

```
L_relational = Σ_stages Σ_{i≠j} similarity(pathway_i, pathway_j)
```

Where similarity could be: cosine similarity of output representations,
mutual information of activations, or CPA overlap (as measured in v7
probes — we already have this metric).

### Cone constraint (guided search)

Each pathway has a cone-shaped region of function space it's allowed
to explore. The cone points toward a northstar (target behavior).

```
                    ★ northstar
                   /|\
                  / | \  aperture (adjustable)
                 / .|. \
                / . | . \
               /  search  \
              ────────────── parameter space
```

**Cone aperture schedule:**
```
Early training:   wide    → pathways explore, find their specialty
Mid training:     narrow  → pathways commit to their function
Late training:    tight   → circuits crystallize, topology freezes
```

**Cone penalty in loss:**
```
L_cone = Σ_pathways max(0, distance(pathway_output, northstar) - aperture)
```

Only penalizes when the pathway drifts outside its cone. Inside the
cone, the pathway is free to explore.

### Ternary flip rate modulated by cone

```
flip_rate ∝ cone_aperture × distance_from_northstar
```

- Far from target + wide cone → high flip rate (explore topology)
- Close to target + narrow cone → low flip rate (crystallize)
- Topology anneals INTO the cone

This generalizes v7's flip rate modulation (flip_rate ∝ r₁) to
per-pathway, per-stage control guided by the cone structure.

### Combined loss

```
L_total = L_task                                # predict correctly
        + λ_rel × L_relational                  # pathways must differ
        + λ_cone × L_cone                       # stay within cone
        + λ_stage × Σ L_per_stage_relational    # stages must differ
```

The relational terms push pathways APART (differentiate).
The cone terms pull pathways TOWARD their northstar (specialize).
Together they carve the sieve — each pathway occupies a unique,
targeted region of function space.

### Why this forms fast

The cone pre-prunes most of the search space. The relational loss
pre-separates pathways. The ternary topology makes decisions discrete.
Combined:

- Without cone: search 3^250M (intractable)
- With cone + relational: search a tiny manifold within the space
- The pathway's target function constrains which topologies are useful
- An "addition pathway" doesn't need connections to comparison circuits
  → those weights go to 0 (disconnected) → the mold carves itself
- Discrete ternary decisions crystallize faster than continuous float
  convergence — there are only 3 choices per weight, not infinite

The sieve shapes should emerge quickly because the training signal
is clean (holographic formal data) and the search is constrained
(cone + relational loss). This is not gradient descent in the dark —
it's guided topology crystallization.

## Sieve Design Questions (for next session)

### Pathway interaction

Do pathways within a stage communicate?
- **Independent:** each pathway is its own transformer. Simple.
  Differentiation relies entirely on relational loss + cone.
- **Cross-pathway attention:** pathways can attend to each other's
  outputs. Richer but more complex. Risk: pathways learn to copy
  each other (defeating the sieve purpose).
- **Shared normalization:** pathways share RMSNorm statistics but
  have independent weights. Mild coupling.

Recommendation: start independent. Add interaction only if pathways
fail to differentiate.

### Sieve → Reducer

How do N parallel pathway outputs merge for reduction to next stage?
- **Concatenate:** 4 pathways × d_model → 4·d_model, then project
  down. Simple but requires a projection layer.
- **Cross-attention pool:** reducer queries attend to all pathway
  outputs. Natural extension of v7's StageReducer.
- **Gated sum:** learned pathway-level gates select which pathway
  contributes to each reduction position. Like feedback gates but
  between pathways and the reducer.
- **Mixture of experts (MoE):** router selects top-k pathways per
  token. Sparse but proven at scale.

### Feedback routing

How does downward feedback route through the sieve?
- **Broadcast:** each pathway receives the same feedback from the
  stage above. Simple. Each pathway integrates it differently
  because of different ternary topology.
- **Routed:** feedback is split across pathways based on content.
  More targeted but requires a routing mechanism.
- **Per-pathway feedback:** each pathway in stage N+1 feeds back
  to a specific pathway in stage N. Creates dedicated channels.

### Northstar definition

What concretely IS the northstar for each pathway?
- **Function prototype:** a target output distribution for known
  inputs. E.g., pathway A's northstar is "given addition inputs,
  produce the sum."
- **Representation target:** a target activation pattern that
  embodies the desired function.
- **Emergent:** no explicit northstar. Relational loss + data
  signal alone shapes the pathways. The cone is implicit in the
  loss landscape, not explicit.
- **Soft labels:** partial supervision. Some training examples
  include a "this is addition" signal that routes to pathway A.

### Pathway count

How many pathways per sieve?
- **4 pathways:** matches the primary operation types
  (arithmetic, comparison, composition, routing/identity)
- **8 pathways:** finer granularity
  (add, subtract, multiply, divide, compare, predicate, compose, route)
- **Per-stage variable:** Stage 1 has many (surface is diverse),
  Stage 4 has few (computation is concentrated)

### d_model per pathway

- **Full width:** each pathway d=1024. Total per stage = 4 × 1024.
  Maximum capacity per pathway. Fewer pathways fit in budget.
- **Split width:** each pathway d=256. 4 pathways. Total per stage
  = same as single d=1024 pathway but more parallel.
- **Tapered:** Stage 1 pathways narrow (d=256), Stage 4 pathways
  wide (d=1024). Matches computation density.
